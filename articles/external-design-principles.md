## 1. 外部設計とは何か

**外から見えるものを全て具体的に定義する工程。**

「外から見えるもの」とは、人間のユーザーだけでなく、APIを呼び出すフロントエンドや外部サービスから見えるものも含む。

要件定義が「何を作るか（What）」を決めるフェーズなら、外部設計は「ユーザー・外部システム（モバイルアプリやフロントエンドなど、自分たちのAPIを呼び出す側）にどう見せるか・どう渡すか（How it looks）」を決めるフェーズ。

```
要件定義
 「注文一覧を表示したい」
 ↓
外部設計
 「どの項目を・どの順で・どんな操作ができるか」を定義する
 ↓
内部設計
 「どのSQLで・どのクラスで実現するか」を決める
 ↓
実装
```

## 2. 外部設計の範囲

外部設計が対象にするのは「外から見える・操作できるもの全て」。

### 画面設計

ユーザーが目にする画面そのもの。表示する項目、配置、タイトルの有無、デフォルトの並び順など。

**例：注文一覧画面であれば**
- 表示する項目（注文番号・顧客名・金額・ステータス・作成日時）
- デフォルトの並び順（新着順 / 登録順）
- ページネーションの有無とページあたりの件数

### 画面遷移設計

どの画面からどこへ遷移するかの定義。画面単体で設計を完結させず、全体の流れを意識する。

### API設計

フロントエンドや外部システムから見えるインターフェースの定義。エンドポイント、リクエストパラメータ、レスポンスの構造がここに含まれる。

```
GET /api/orders

レスポンス:
{
  "orders": [
    {
      "id": 1,
      "customer_name": "山田太郎",
      "amount": 3200,
      "status": "配達済み",
      "created_at": "2026-06-04"
    }
  ],
  "total": 100,
  "page": 1
}
```

このレスポンス構造は呼び出す側に丸見えである。画面の表示内容がユーザーに見えるのと同じ構造だ。

### バリデーション・エラー設計

入力が正しくない場合や、システムエラーが発生した場合にユーザーに何を見せるかの定義。エラーメッセージの内容・表示タイミング・見え方がここに含まれる。

### データ出力設計

CSVやExcelなど、データをファイルとして出力する際の定義。どの項目を・どの形式で出力するかはユーザーに見えるため、外部設計の範囲になる。

### 権限設計

誰が何を見られるか・何ができるかの定義。ユーザーの立場によって画面や機能の見え方が変わる場合、それを明示的に定義する。

| ポータル | 見える機能 |
|---|---|
| 管理者ポータル | 全機能 |
| 一般ユーザーポータル | 限定された機能 |
| ゲストポータル | 閲覧のみ |

## 3. 外部設計と内部設計の違い

境界線は一つ：**ユーザー・外部システムに見えるかどうか。**

並び順を例に取ると、その違いが明確になる。

| フェーズ | 定義内容 |
|---|---|
| 外部設計 | 「新着順（作成日時の降順）で表示する」 |
| 内部設計 | 「`created_at DESC` でSQLクエリする」 |

「新着順で表示する」はユーザーが目で確認できる振る舞いなので外部設計。「どのSQLで実現するか」はユーザーには見えないので内部設計。

内部設計は外部設計が定義した契約を守りながら、実装方法は自由に選べる。

---

## 4. 誰のための外部設計か

外部設計書を読む人は、開発者だけではない。

| 読む人 | 目的 |
|---|---|
| 実装する開発者 | 地図として。設計書なしに実装を始めることは、地図なしに目的地へ向かうようなもの |
| テストするテスター | 「正しい状態」の定義として。何が正解かわからなければ、テストができない |
| クライアント | 「これで合ってますか？」の確認対象として |
| 将来引き継ぐ開発者 | 機能の説明書として。会ったことのない人にシステムを引き渡す唯一の手段 |

---

## 5. なぜ外部設計が必要か

外部設計は、**修正コストが低い段階で確認できる最後のチェックポイント**である。

| 気づくタイミング | 修正コスト |
|---|---|
| 要件定義段階 | 低い（会話で修正できる） |
| **外部設計段階** | **低い（設計書の修正で済む）** |
| 実装後 | 高い（作ったものを壊して作り直す） |
| リリース後 | 非常に高い（ユーザー影響・信頼損失も含む） |

外部設計書をクライアントに確認してもらうフェーズを設けることで、「実装してから違うと言われる」を防げる。設計書を修正するコストは、コードを作り直すコストより圧倒的に低い。

外部設計のレビューを省略して先に実装を見せるやり方は、スピード感があるように見えて、**間違った方向で実装するリスク**を後ろに先送りしているだけになりうる。

## 6. エッジケースの見つけ方

正常に動いているとき（正常系）だけを設計しても不十分。「誰も言わないまま残りやすいケース」こそ、後からトラブルになりやすい。

> 誰も言わないケースの典型例：データが0件のとき、エラーが起きたとき、権限のないユーザーがアクセスしたとき。

### パターンアプローチ

全てのエッジケースをゼロから洗い出すのは現実的ではない。**カテゴリを先に決め、各カテゴリの状態を洗い出す**2段階のアプローチが有効。

| カテゴリ | 状態の例 |
|---|---|
| データ量 | 0件 / 少ない / 大量 |
| ユーザーの権限 | 管理者 / 一般ユーザー / ゲスト / 未ログイン |
| システムの状態 | 正常 / サーバーエラー / 読み込み中 / タイムアウト |
| 入力値 | 正常 / 不正な形式 / 空 / 最大文字数超過 |

カテゴリはパターン化できる。各カテゴリのどの状態を定義すべきかは、システムの特性とリスクで判断する。

## 7. 要件定義との繋がり

要件定義が終わると、外部設計のインプットが揃う。要件定義が「何を作るか」を決めているのに対し、外部設計はその要件を具体的な振る舞いに落とし込む。

要件には書かれていない多くの決定事項が、外部設計の段階で初めて問われる。

**例：「注文一覧を表示したい」という要件に対して外部設計が決めること**

| 問い | 決定が必要な理由 |
|---|---|
| どの項目を表示するか | 要件には書かれていない |
| デフォルトの並び順は何か | 言わなければ開発者の解釈に任される |
| ページネーションは必要か | データ量によって変わる |
| 検索・絞り込みはできるか | ビジネスの都合による |
| データが0件のとき何を表示するか | 誰も言わないまま残りやすい |

並び順の問題は、要件定義ドキュメントでも「暗黙の前提」として登場した。外部設計で明示することで、同じトラブルを設計段階で防げる。

### ドメイン知識が必要な理由

要件を外部設計に落とし込む際、要件に書かれていない多くのことを決めることになる。その材料として必要なのがドメイン知識。

「この業務フローなら、この順で並ぶのが自然」「この機能には権限制御が必ず必要になる」という判断は、ドメイン知識があって初めてできる。

## 8. クライアントとの合意の境界線

外部設計の決定事項には、開発者が判断していいものと、クライアントと合意が必要なものがある。

| 判断の種類 | 誰が決めるか |
|---|---|
| ビジネスルール・業務上の判断が必要なもの | クライアントと合意 |
| 技術的なベストプラクティスの範囲のもの | 開発者が判断 |

「検索機能が必要かどうか」は技術的な判断に見えるが、「一覧は常に20件以下だから不要」というビジネス上の判断が絡む。機能の有無を含め、ユーザーの体験に影響するものはクライアントとの合意が必要になる。

---

## 9. ツールと状況

外部設計のツールに「これが正解」はない。状況によって適切なアプローチが変わる。

| 状況 | 向いているアプローチ |
|---|---|
| 既存システムへの機能追加・改修 | Markdown や簡易メモで軽く定義 |
| 完全新規の画面・機能 | ワイヤーフレームや図で先にイメージを合わせる |
| 複数の開発者が同じ機能に関わる | 正式な設計書で共通認識を作る |
| クライアントと頻繁に会話できる | 軽量なもので十分 |
| クライアントとのやり取りが少ない | より丁寧な文書が必要 |

ツールの選択より重要なのは**目的**。外部設計のゴールは「実装前に全員のイメージを合わせること」であり、それを達成できれば手段は問わない。

大規模なチームや組織で働く場合、複数人・複数チームが同じシステムに関わるため、軽量なアプローチでは認識のズレが生まれやすくなる。チームの規模が大きいほど、設計書の精度と形式が重要になる。

ツールや形式がどうあれ、外部設計は設計の内容を具体的な成果物（設計書・図・仕様書など）として文書化することまでがセット。何を作るかはプロジェクトの規模や状況によって変わるが、「頭の中で決めた」で終わらせないことが重要。

## 10. 内部設計への引き渡し

外部設計が内部設計に渡すのは、「**外側から見た振る舞いの契約**」。

- この画面にはこの項目をこの順で表示する
- このAPIはこのリクエストを受け取り、このレスポンスを返す
- この操作をしたとき、このエラーメッセージを表示する

内部設計は、この契約を守ることを前提に、実装方法を自由に選べる。外部設計が「何を実現するか」を決め、内部設計が「どうやって実現するか」を決める。

契約として明文化されていない部分は、開発者の解釈に任される。外部設計の曖昧さは、そのまま実装のズレになる。

### DB設計との関係

DB設計は内部設計の範囲。テーブル構造やカラム名はユーザーに見えないため、外部設計の対象にはならない。

ただし、外部設計で「何を表示するか」が決まることで、DBに何を持たせるべきかが決まる。外部設計がDBを直接定義するのではなく、表示するデータの要件が内部設計（DB設計）へのインプットになる。

```
外部設計
 「注文一覧に顧客名・金額・ステータスを表示する」
 ↓ 必要なデータが決まる
内部設計（DB設計）
 「注文テーブルにcustomer_name・amount・statusカラムが必要」
```

## まとめ

- 外部設計の本質は「外から見えるものを全て具体的に定義すること」。ユーザーや外部システムに見える・操作できる全てのものがその対象になる。
- 範囲は画面・画面遷移・API・バリデーション・データ出力・権限設計。全て「外から見える」という一本の軸で繋がっている。
- 外部設計と内部設計の境界線は「ユーザー・外部システムに見えるかどうか」。並び順のような振る舞いは外部設計、それをどのSQLで実現するかは内部設計。
- 外部設計書を読むのは開発者だけでなく、テスター・クライアント・将来の引き継ぎ者も含む。複数の人間の共通言語になる。
- 外部設計は修正コストが低い段階で確認できる最後のチェックポイント。クライアントレビューを省略することは、リスクを実装後に先送りすることになる。
- エッジケースはゼロから洗い出さず、カテゴリ（データ量・権限・システム状態・入力値）を先に定め、各状態を洗い出すパターンアプローチが有効。
- 外部設計の決定事項には、クライアントと合意すべきものと開発者が判断できるものがある。ビジネスの判断が必要なものは必ずクライアントと合意する。
- ツールは状況次第。重要なのは「実装前に全員のイメージを合わせる」というゴールを達成することであり、手段は目的に合わせて選ぶ。
